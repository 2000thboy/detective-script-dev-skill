# Demand Clarification Protocol (需求澄清协议)

## 核心原则

- 在用户明确确认用中文创作之前，**绝不创建任何 .md/.json 文件**
- 讨论是分层进行的，每层聚焦一个维度
- 用户可以随时说"返回上一层"回到上一讨论层
- 每层确认后才进入下一层
- 诡计相关讨论可能反复迭代，允许多次返回修改

## 返回机制

- 每层结束时问："确认进入下一层，还是返回上一层修改？"
- 支持的返回指令：
  - "返回上一层"
  - "回到 Layer 1/2/3"
  - "修改 Layer X 的 XXX"
  - "重新讨论诡计"
- 必须记住当前所在层，返回后从该层重新开始

---

## Layer 1: 基础定位

讨论内容：
- 同人 or 原创？如果是同人，原作是什么？角色限制？
- 参考作品（3部以内）
- 禁忌元素（绝对不能出现的内容）
- 目标字数范围
- 预计章节数
- 核心诡计类型偏好（密室/不在场证明/叙述性诡计/社会派/其他）

确认后才进入 Layer 2。

---

## Layer 2: 结构偏好

讨论内容：
- 诡计类型偏好（确认/细化 Layer 1 的选择）
- 节奏偏好（慢热铺垫型 / 快节奏推进型）
- 大纲深度偏好（详细到每章节拍 / 粗略到章节主题）
- 章节长度目标
- 反转次数预期
- 是否接受多线叙事

确认后才进入 Layer 3。

---

## Layer 3: 风格偏好

讨论内容：
- 文笔风格（轻小说/纯文学/社会派/本格）
- 角色关系基调（友情/竞争/暧昧/疏离）
- 对话密度偏好（高对话 / 高描写 / 平衡）
- 内心独白偏好（第一人称大量独白 / 第三人称客观 / 混合）
- 时代背景偏好（现代/昭和/古代/架空）
- 是否需要异文化展示

确认后才进入 Layer 4。

---

## Layer 4: 确认汇总

- 打印前三层所有选择的中文汇总
- 明确告知："以上信息确认无误吗？确认后将创建 case 并开始创作。"
- 只有用户明确说"确认/是/OK/没问题"后才执行 `case init`
- 如果用户说"修改XXX"，返回对应层

---

## 诡计迭代特别规则

当讨论进入诡计层面时：
- 允许多次在 Layer 1-4 之间跳转
- 诡计讨论可能产生多个候选方案，需逐一讨论后选择
- 诡计确认后，视为 Layer 4 的一部分，需最终确认
- 诡计相关的返回指令："重新讨论诡计"直接回到 Layer 1 的诡计类型讨论
